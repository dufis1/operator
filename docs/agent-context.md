## Hard-Won Knowledge (read before touching relevant code)

- **`rich.live.Live` + `Prompt.ask` don't compose; reach for `readchar` and a callable right-pane instead** (session 120). Building the picker primitive surfaced the architectural fork: Rich's `Prompt.ask` writes to the console synchronously and Live owns its render region, so they fight over the cursor + cause flicker if you try to host text input inside a Live block. The clean split is: picker phases own the screen via Live and read raw keys via `readchar.readkey()` (returns `\x1b[A` for arrows, `\n`/`\r` for enter, ` ` for space — handle both `\n` and `\r` for cross-platform Enter). Text-input phases (custom-mode name prompts, API-key prompts) drop the Live region entirely and use `Prompt.ask` normally. Trying to keep the build card "always visible" through text-input phases is a maze for very little gain — the picker phases are where the live vibe pays off; text input is short and sequential. Second design lever: the picker's right pane needs to support both per-choice previews (step 1 portrait gallery — preview is a property of the highlighted choice) and global state previews (steps 2/3 build card — preview reflects toggled state across the whole list, doesn't depend on cursor). One-knob solution: a `right_pane: Callable[[cursor, checked], Renderable]` argument that wins over per-choice `Choice.preview` when provided. The wizard passes `right_pane = lambda c, ch: state.card(mcps=names_where(ch))` — closes over global wizard state and re-renders on every keystroke without mutating state during render.
- **Rich markup parser swallows `[x]` silently — use non-ASCII glyph in brackets** (session 119). Rendering a checked-state marker in toggle lists with `[x]` made it render as a blank because Rich treats `[x]` as a (non-existent) style tag and strips it. The asymmetry only showed up when some items were default-checked; `[ ]` (with a space) printed literally because " " isn't a valid style name. Fix: switch to `[✓]` — non-ASCII content inside brackets isn't parsed as a style. Same applies to any character that *could* be a style name (`[u]`, `[i]`, `[b]`, `[red]`, etc). Reach for a glyph the parser won't try to interpret, or escape with `\\[` if you really need ASCII brackets.
- **`MeetingRecord.tail()` must scope to the current session or the LLM echoes prior runs' replies and stops calling tools** (session 117). `MeetingRecord` is append-only JSONL keyed by meet slug — same URL ⇒ same slug ⇒ same file. When a user reruns `brainchild <name> <url>` against the same meeting (common during development), the file accumulates every prior session's entries. `tail(n)` was originally implemented as "last n lines of the file regardless of origin," which meant `LLMClient._tail_messages()` replayed the agent's own prior assistant replies back to it as `role="assistant"` history. Downstream effect: the model pattern-matched on its prior outputs and replied directly instead of calling the MCP tool that originally produced those outputs. Worst case observed in diagnosis: user said "mojo jojo" → bot replied `Created https://linear.app/mojo-jojo/issue/MOJ-11/fold-my-laundry` with **zero tool calls logged** — it hallucinated the URL from MOJ-N patterns in prior replies. The tell is extremely easy to miss: fresh-slug runs (`brainchild <name>` with no URL, which hits meet.new and gets a fresh slug) work fine; only same-URL reruns regress. Symptom from the logs is `len(messages)=20/22/...` on the problematic runs vs `len(messages)=1` on the fresh-slug run. Fix: write a `{"kind": "session_start", "timestamp": ...}` boundary marker on every `MeetingRecord.__init__`; `tail()` scans back from the end of the parsed entries to the most recent marker and returns only entries after it. Raw JSONL preserves the full audit (you'll see two `"session_start"` lines in the file after a rejoin). Don't "fix" this by purging old history — the file is valuable as a durable record of the meeting; only the *LLM-visible window* needs to be session-scoped. Tests in `tests/test_chat_hardening.py::test_meeting_record_tail_roundtrip` cover the reopen-yields-empty-tail contract.
- **Startup banners must print before log output or they get buried; don't wait for MCP status if the loadout names are enough** (session 117). First attempt at the 15.5.4 startup banner called `_print_startup_banner(mcp, skills)` *after* `mcp_thread.join()` so the loadout line could show per-server ✓/✗ status. Live-verified on `brainchild pm`: the banner printed correctly but sat 40+ lines down in scrollback, below MCP tool-discovery chatter + `TIMING` lines + `meet.new resolved to ...` → user's first reaction was "where's the banner?" - **Playwright `sync_playwright.__exit__` can wedge forever on a dead CDP socket; removing `browser.close()` isn't sufficient; SIGKILL the Node driver to unblock it** (session 116). With `launch_persistent_context(headless=False, args=[..., "--headless=new"])`, Chrome sometimes self-exits a beat after the Leave-call click — the CDP socket dies but Playwright doesn't notice. Any subsequent teardown that sends a CDP request (explicit `browser.close()`, OR the context-manager exit's `playwright.stop()`) waits for a response that never arrives. Stack signature to recognize: `greenlet_main` → `_loop.run_until_complete(connection.run_as_sync())` → asyncio `_run_once` → `KqueueSelector.select` → `kqueue.control(None, ...)` with `timeout=None` (no scheduled work, waiting on I/O forever). Rough guide to reproducibility: flaky — roughly 1 in 3 tool-call-path shutdowns hit it, clean-path shutdowns almost never do. First instinct was "remove `browser.close()` since `sync_playwright.__exit__` handles teardown" — that helps the clean path but the same wedge signature reproduces inside `__exit__` on the tool-call path. The fix that actually works: force-kill the Playwright Node driver child process directly — that closes its socket, which unblocks the greenlet's `kqueue.select(None)`, which lets the asyncio loop exit, which lets the context-manager unwind. Implementation in `connectors/macos_adapter.py`: `leave()` does `_browser_thread.join(timeout=2)` → if still alive, `_kill_playwright_drivers()` (pgrep `-P <pid>` + match cmdline on `playwright` + `run-driver` + SIGKILL) → re-join 2s. Don't try to rely on Playwright's graceful teardown path when the CDP socket might be dead — just kill the driver.
- **ChatRunner's main-thread loop needs `_stop_event` guards after any blocking call to avoid a stray post-shutdown iteration** (session 116). `ChatRunner._loop()` runs on the main thread; SIGINT's shutdown handler runs on the main thread too, synchronously (setting `_stop_event`, then calling `connector.leave()` which blocks until the browser tears down). While the handler is running, the loop is frozen wherever it was — often inside `read_chat()` or `get_participant_count()`'s `result_q.get(timeout=N)`. When the handler returns control to the loop, the blocked call resumes and returns its result. The loop body then runs through to completion (logging participant count changes, etc.) *before* the top-of-while check re-reads `_stop_event`. Symptom: a stray `ChatRunner: participant count changed X → 0` log appearing several seconds *after* `MacOSAdapter: left meeting`, attributed to the drained queue returning 0. Fix: `if self._stop_event.is_set(): break` immediately after any blocking call in `_loop` (`read_chat`, `get_participant_count`) so the loop bails before processing results from a mid-shutdown unblock.

Symptom: first smoke-test of `brainchild setup` showed bundled skills as `1. · alpha` instead of `1. [x] alpha`. Fix: use a non-bracket glyph for the "on" mark — `[✓]` for checked, `[ ]` for unchecked. Alternative would be `Console(markup=False)` but that kills other useful formatting. Rule of thumb: anything enclosed in `[…]` that isn't a valid style name is at risk; prefer Unicode glyphs for user-facing status indicators in Rich output.
- **Edit-in-place bundle rewrites must seed the tempdir with the existing bundle first, or unknown files get wiped (session 119).** The `brainchild setup` wizard uses atomic write — build into `agents/.<name>.tmp-<ts>/`, then `os.rename` it over `agents/<name>/` (with a `.bak-<ts>` swap for recovery). First implementation built the tempdir from scratch with only the wizard-owned files (`config.yaml`, `portrait.txt`, `skills/`, stub `README.md`), then swapped. Result: `.env.example`, hand-written README edits, delegate scripts, and any other non-owned siblings in the preset bundle got nuked silently on reconfigure — the user never consented to that. Fix: in `edit` mode, seed the tempdir via `shutil.copytree(target, tmp, dirs_exist_ok=True)` *before* any writes so the full existing bundle carries through the swap. Then selectively overwrite owned files, wipe `skills/` wholesale (skills are re-authored each run), and write `README.md` only if absent. The test `test_edit_in_place_swap_cleans_backup` pins this: asserts `.env.example` + pre-existing README survive, stale `skills/deselected/` is gone, no `.bak-*` lingers on success. Generalizable rule: atomic-write flows that "replace a directory" must preserve any files the operation doesn't explicitly own.

## LLM Interaction Principles

Design patterns for working with LLMs in this product. Read before making architectural decisions about context management or model behavior.

- **Archive-with-metadata pattern:** When removing content from the model's context (tool results, history turns), always replace it with a placeholder that names (1) what happened and (2) what the model can do about it. Example: `[tool result archived — call the tool again with a narrower scope to retrieve more]` is better than silent deletion or `[result omitted]` (which implies error). The model reasons better when it understands its own state and has a recovery path. Applies anywhere context is managed: history trimming, tool result truncation, overflow recovery.

## Phase 11.4 — Skill Loading (design notes)

**History (session 102).** Originally split as 11.4 (generic flat-`.md`) + 11.5 (Claude Code interop on top). Collapsed into a single phase: adopting `SKILL.md` shape + progressive disclosure + `allowed-tools` filtering up front is simpler than shipping twice. `skills.local.yaml` deferred to post-MVP.

Pitch: **"Claude Code in Google Meet."** Users bring their existing Claude Code skills into Brainchild instead of recreating them.

**Source of truth.** Claude Code skills live at `~/.claude/skills/<name>/SKILL.md` (user-level) and `.claude/skills/<name>/SKILL.md` (project-level). Each skill is a folder with a `SKILL.md` — YAML frontmatter (`name`, `description`, optional `allowed-tools`) plus a markdown body. Format is designed for progressive disclosure: only `name + description` loads up front; the full body is pulled in when the model decides the skill is relevant.

**Load model: reference, not copy.** `config.yaml` stores paths (`~/.claude/skills/<name>` or absolute paths), not file contents. On startup, Brainchild reads `SKILL.md` live from the user's machine. Rationale: each user's `~/.claude/skills/` is already their curated, personal set — forcing a copy into the repo would be double-bookkeeping, and the open-source story is "each user makes Brainchild theirs." Downside acknowledged: paths only exist on that user's machine, so a teammate who clones the repo will hit missing skills. Mitigation: log + skip, don't crash.

**Implementation shape.**
- `config.yaml` gains a `skills:` list. Entries are paths (absolute or `~`-expanded). An entry may be either a single skill folder (contains `SKILL.md`) or a parent directory (scanned one level deep for `*/SKILL.md`). Parent-dir scanning is what makes `~/.claude/skills/` a single-line config entry.
- On startup, resolve each path, parse `SKILL.md` frontmatter with `pyyaml` (already a dep).
- **Progressive disclosure.** Inject only `name + description` of each skill into the system prompt as an available-skills list. Register a synthetic `load_skill(name: str)` tool alongside MCP tools. When the model calls it, Brainchild returns the full body of the named skill as the tool result. Model then re-plans with the body in context. Matches Claude Code's loading behavior; keeps token cost flat as users accumulate skills.
- Missing path → WARN log + skip. Do not crash startup.
- Malformed `SKILL.md` (bad YAML, missing `name` / `description`) → WARN log + skip.
- Duplicate skill names across paths → last one wins, WARN on collision.
- Startup log line: `SKILLS: N/M loaded (<name> ✓, <name> ✓, <name> ✗ missing)`. Mirrors the MCP startup banner pattern from session 87.
- **Deferred to post-MVP:** optional gitignored `skills.local.yaml` for personal overrides — teams share a baseline `config.yaml` and layer personal skills on top. Not MVP-critical.
- System prompt ordering: base prompt → skills block → MCP hints → MCP status (most dynamic last, so skills share a stable prefix for prompt caching).

**Tool-surface filtering.** Most interesting Claude Code skills assume tools Brainchild doesn't have in a meeting context. Split by `allowed-tools`:
- **Pure-instruction skills** (no `allowed-tools`, or only MCP-backed tools): ship as-is. Tone guides, commit-message style, decision frameworks, interview rubrics.
- **MCP-backed skills** (Linear, Gmail, Calendar): work once the corresponding MCP server is configured.
- **Tool-heavy skills** (`Bash`, `Edit`, `Write`, `Read`): known to misfire. Five common ones that break: `/commit`-style git skills, the `pdf` skill, the office-suite skill (`docx`/`xlsx`/`pptx`), PR-review skills (`/review-pr`, `simplify`), refactor/debug-test-failure skills. Common thread: **anything that assumes a filesystem, a shell, or the Read/Edit/Write triad** breaks in a Meet context.
- On import, check `allowed-tools`; if it references tools Brainchild can't honor, log a WARN with the specific unsupported tools. Still load the skill (the model may decline to invoke it in the meeting context) but the user knows why it might misfire.

**Out of scope for 11.4.** Skill *execution* beyond injecting body text — i.e., we are not implementing `Bash`/`Edit`/`Write`/`Read` to make tool-heavy skills actually work. Pitch is honest: "bring your instruction skills." Also out of scope: hot-reload when a `SKILL.md` changes on disk mid-meeting — load at startup only (restart required for changes).

## Phase 15.5 — Opinionated Quickstart (design notes)

Pitch: **"Choose your fighter → add power-ups → go."** The agents gallery is the gallery layer; the wizard turns it into a 3-step setup. Implicit differentiator vs. joinly's framework-shaped positioning.

**Why a gallery, not forks.** Forks fragment git history and don't make the parent repo look alive. A `roster/` folder with PR contributions keeps everyone on main, makes contributors visible in the commit log, and turns every seeded-persona build into a permanent artifact. This is the primitive; forks were rejected explicitly. Reference patterns: Ollama model library, LangChain templates, n8n templates, ComfyUI workflows, Vercel templates, Hugging Face Spaces.

**Folder format (already locked in `roster/README.md`).** Each roster member is a self-contained folder with a runnable `config.yaml`, optional `skills/` directory, a short `README.md` (what / who / needs / setup / demo), and `.env.example`. The roster member must run on a fresh clone after the user fills in keys — no hidden dependencies.

### 15.5.1 — `claude-code` starter agent

Canonical example. Demonstrates the format and is the launch hero. **Depends on Phase 11.5 (Claude Code skill import) being complete** — without skill import, this agent has nothing distinct to show.

- `roster/engineer/config.yaml` — complete Brainchild config; either OpenAI or Anthropic as default (decide near build time, lean Anthropic since the framing is "Claude Code"); `mcp_servers` includes Linear + GitHub; `skills:` points at `~/.claude/skills/` by default, with a comment explaining how to swap to a custom path.
- `roster/engineer/README.md` — "What it does: brings your existing `~/.claude/skills/` into Google Meet so the agent files Linear tickets, looks up GitHub PRs, and follows your team conventions during a live call." Setup: copy `config.yaml`, fill `.env`, run. Demo: short GIF of an artifact appearing in chat mid-sentence.
- `roster/engineer/skills/` — leave empty by default. The whole point is the user's own skills get loaded from `~/.claude/skills/`. Optionally bundle one demo skill that's safe to ship (e.g. `file-linear-ticket.md`) so the agent has at least one skill even if the user has none locally.
- `.env.example` — `OPENAI_API_KEY` or `ANTHROPIC_API_KEY`, `LINEAR_API_KEY`, `GITHUB_TOKEN`.

### 15.5.3 — Runtime architecture (roster as source of truth)

**The architectural shift** (session 113): today, `python __main__.py <url>` reads an ambient `config.yaml` at repo root — "whatever was last copied there." Replaced by explicit per-run bot selection. Every run names the bot.

- Positional CLI in `__main__.py`:
  - `brainchild <name> <url>` — run that bot in that meeting. Load `roster/<name>/config.yaml` directly.
  - `brainchild <name>` (no URL) — auto-open `meet.new`, capture slug, join as that bot. (Absorbs old 15.5.4's `meet.new` deliverable.)
  - `brainchild setup` — launches the wizard (see 15.5.5).
  - `brainchild list` — prints available roster members + one-line pitches.
  - `brainchild` (bare) — prints usage + roster list.
- Dispatch rule: if first arg matches a `roster/*` folder, treat as bot-name; if it's a known subcommand (`setup` / `list`), dispatch there; else error with helpful usage.
- Delete root `config.yaml`. Config loading reads `roster/<name>/config.yaml` at runtime.
- `.env` stays at repo root, shared across all bots (keys like `ANTHROPIC_API_KEY` are environment-scoped, not bot-scoped).
- Ship `./brainchild` bash wrapper in repo root: `#!/usr/bin/env bash` + `cd "$(dirname "$0")"` + `source venv/bin/activate` + `exec python __main__.py "$@"`. Provisional launcher until product name is locked and curl installer ships post-launch.
- **Breaking change:** every doc, test, and reference to `python __main__.py <url>` migrates to `brainchild <name> <url>`. Pre-launch, so the cost is grep + update, not deprecation.

### 15.5.4 — Face system + profile banner

Each bot has a unique pixel-art face that prints on every run. Faces are the bot's identity and the "you're playing as X" moment when a meeting starts.

- **Deterministic face generator.** Half-block Unicode (`▀ ▄ █`) + ANSI 256-color. 16×20 pixels — fits 3-across in a standard terminal, readable. Seed = hash(bot name), so the same name always produces the same face. Parts library: eyes, mouth, brow, hair/hat, accessory, palette.
- **Shipping portraits.** Run the generator for `engineer` / `pm` / `designer`, hand-curate until good, commit as `roster/<name>/portrait.txt`.
- **First-run write hook.** Any `brainchild <name>` run that finds no `portrait.txt` in the roster folder generates one and writes it. This is the "gift" for contributors: they add `roster/translator/` with a config and README, run `brainchild translator` once, and a face file materializes for them to commit with their PR.
- **Startup banner.** Every `brainchild <name>` run prints a block before joining:
  ```
  ╔════════════════════════════╗
  ║   <face>      ENGINEER     ║
  ║               looks up GH  ║
  ║               MCPs: GitHub ║
  ║               Skills: 4    ║
  ║               Model: ...   ║
  ╚════════════════════════════╝
  ```
  Reads from the active config — pulls agent name, one-line tagline (from README h1 or a config field), MCP names, skill count, model.
- **`--plain` fallback.** Prints ASCII-only banner for screen readers / hostile terminals / CI logs.
- Dep: `rich` for color/boxes/layout (new dep — already battle-tested, 1-package install).

### 15.5.5 — `brainchild setup` wizard (creation flow)

**Key reframe** (session 113): wizard is no longer "activate a bundle by copying config to root." It's a *creation* tool — builds a new `roster/<name>/` folder so the user ends up with their own bot they run explicitly.

- `brainchild setup` — four steps:
  1. **Choose a template (or start blank).** Fighter-select grid UI — colored boxes, pre-shipped faces visible, numbered prompt. "Start blank" produces a minimal config stub.
  2. **Power-ups — awareness + toggle.** Show the template's bundled MCPs and skills as a checklist (including commented-out power-up entries in the template config). User toggles what ends up in the new bot. Narrow scope: only touches what the template declares. Out of scope: arbitrary MCP discovery, paste-a-URL flow, `~/.claude/skills` path editor (all post-launch).
  3. **Keys.** Prompt for any env vars the new bot needs that aren't already in `.env`. `getpass` for secrets (no echo, never print existing values). Merge into `.env`, preserving unrelated keys. Atomic write.
  4. **Finalize.** Atomic-write the new `roster/<new-name>/` folder: `config.yaml` (built from template + toggles), `README.md` stub, `.env.example`. Generate face via 15.5.4's generator, write `portrait.txt`. Print: `Your bot is ready. Run: brainchild <new-name> <meet-url>`.
- Re-run-safe: name-collision check against existing roster folders.
- Intent of step 2: pure FYI — user should *know* what their bot can do (so they know to ask it to file a Linear ticket), not be forced into configuration work. Checkbox ergonomics, not a forms-based MCP configurator.
- **Depends on 15.5.3** (for `setup` dispatch surface) and **15.5.4** (for face generator + pre-baked shipping portraits rendered in step 1's grid).
- ~250 lines of Python + `rich`. Out of scope: voice, MCP OAuth flows, model selection UI beyond what the template dictates. Those land in the post-MVP full wizard.

Real users browsing the repo see contributor activity on `main` — social proof as code. Ship 3 roster members at launch (engineer + PM + designer), target 5 within week 1 via seeded PRs.

## Hard Won Knowledge

*Append-only log of bugs that were hard to diagnose. Each entry: what the bug was, what made it tricky, how it was fixed.*

- **`yaml.safe_dump` silently eats comments, blank lines, and hand-formatting.** (Session 121.) The setup wizard's edit-in-place mode runs `yaml.safe_load(existing_cfg)` → user edits → `yaml.safe_dump(cfg)` → writes back. Comments, section dividers, inline explanatory notes, and custom quoting all vanish. Surfaced when the user smoke-tested the wizard on `agents/{pm,designer,engineer}/config.yaml` — `git status` showed huge diffs even on presets that didn't change any wizard inputs, just because `yaml.safe_dump` round-trips drop everything the parser doesn't care about. What made it tricky: the wizard behaves correctly (values round-trip intact) so no test would catch it; you only see the damage in `git diff`. Mitigation for now: restored configs via `git checkout --`. Proper fix (deferred): swap to `ruamel.yaml` with `YAML(typ='rt')` — round-trip loader preserves comments and formatting. Or, only rewrite the specific fields the wizard touches (`enabled` flags, `skills.paths`) via a targeted string/diff approach. Not blocking; worth addressing before inviting external contributors who'll keep editing their configs by hand.
- **Rich markup parser swallows `[x]` in printed text.** (Session 119.) First draft of the power-ups toggle rendered checked items as `[x]` and unchecked as `[ ]`. `[ ]` printed literally (invalid markup), but `[x]` vanished entirely — Rich interpreted it as a (non-existent) style tag and stripped it. Took a minute to spot because the asymmetry only appeared when some items were default-checked. Fix: switched to `[✓]` (non-ASCII bracketed content isn't parsed as a style tag). Generalizable lesson: any `[letter]` or `[word]` in Rich-printed strings is at risk of being treated as markup — either escape with `\[` or use unicode glyphs.
- **Rich `Live` can't host `Prompt.ask`/`input()` mid-render.** (Session 121.) First attempt at step 3's "add your own skills" UX tried to pop a text input when a sentinel row was toggled mid-picker. Doesn't work — the `Live` region owns the terminal until exited, and input prompts don't compose with it. Fix: either exit the Live first (picker confirms → then prompt), or render a "queued action" hint and act on it after the picker closes. Final design landed on the former — `Confirm.ask` after the picker closes, path-input loop if confirmed.
- **Right-pane column squeezes inline sublabels past terminal-width truncation.** (Session 121.) Picker rows rendered `label <spaces> sublabel` on one line in a `Table.grid` left column; a 40-col build card in the right column left only ~32 cols for the left, and Rich silently truncated the dim sublabel text with no ellipsis. Looked like a 60-char cap bug at first because the skill subtitle *also* had a `desc[:60]` truncation in the code, which was a red herring — removing the slice didn't fix the visible cut-off. Fix: move sublabels to their own indented line under the label. No more horizontal-width race.
- **Rich `Panel.border_style` is a single value — no per-char frame colors.** (Session 122.) User asked for a rainbow frame on the build card. `Panel` accepts one `border_style` and applies it uniformly; there's no hook to vary color per character. Fix: drop `Panel` entirely and hand-compose the frame. Build top/bottom/side chars as raw strings, wrap each character with a fresh ANSI color + reset, join into a single string, and feed to `rich.text.Text.from_ansi` so Rich re-parses the sequences into styled segments at render time. Worth knowing because the obvious "style the Panel border" path is a dead end for any gradient/rainbow/per-char frame effect.
- **`len()` lies about cell width for wide glyphs (⚡, emoji).** (Session 122.) When hand-composing the rainbow card, naive `len(line)` padding misaligned the power-ups row because `⚡` is 2 terminal cells wide, not 1. Fix: `from rich.cells import cell_len` (or use `wcwidth`) for cell-accurate padding. Lesson applies any time you hand-compose ANSI-framed output instead of letting Rich's layout engine handle spacing.
- **`rich.prompt.Prompt.ask(..., default="")` renders an empty `()` in the label.** (Session 122.) The prompt text showed as `tagline (one line, under ~60 chars) ():` — Rich always appends `(default_value)` to the label when a default is set, and an empty-string default renders as an empty pair of parens. Fix: pass `show_default=False`. Non-obvious because the default value *itself* is correct and everything works — it's just cosmetic noise in the label.
- **`claude -p` in a non-interactive subprocess defaults to permission mode `ask` — which denies every Write/Edit/Bash.** (Session 125.) `delegate_to_claude_code` MCP spawns `claude -p <task> --worktree` as a stdio subprocess. In TTY-less contexts `ask` mode has no human to confirm with and auto-denies, so every file edit inside the worktree silently fails — the subprocess exits "success" but the tool list is effectively read-only. Surfaced only once we test-drove the harness: Engineer kept reporting "I wasn't able to write that file" for trivially-writable tasks. Fix: pass `--permission-mode bypassPermissions` explicitly (safe because Brainchild already confirm-gates each delegate call and `--worktree` throwaway-isolates the fs). Made the mode configurable via `DELEGATE_PERMISSION_MODE` env var in `agents/engineer/config.yaml → mcp_servers.delegate.env` so users who want a tighter sandbox can pick `acceptEdits` (file ops yes, Bash no). Lesson: whenever an LLM CLI is spawned as a subprocess and behaves like it has fewer tools than expected, check the permission mode first — it's almost always the culprit and the symptom (silent no-op) looks like anything but a permission issue.
- **LLM routes local-path questions to GitHub MCP when it has both `github__*` and `delegate` tools available.** (Session 125.) Engineer's first harness session: asked "how many python files are in pipeline/?", model called `github__search_code(query='path:pipeline/ extension:py', repo='<guessed>/<guessed>')` — hit 404s, didn't recover. It has no way to know "this path is local not remote" from tool schemas alone; github's description sounds plausible for any repo-ish question. Fix: add an explicit tool-routing paragraph to the bot's `llm.system_prompt`: `github__*` is for *remote* repos the user names by owner/repo; any local path = delegate to Claude Code; "when in doubt, prefer delegate." Generalizable: whenever two tools have overlapping surface area (both can "read code"), you need a routing rule in the prompt — tool descriptions alone aren't enough signal.
- **`ChatRunner` treats `connector._connected = False` as an unexpected disconnect.** (Session 125.) First `TerminalConnector` draft implemented `/quit` by flipping `_connected` to `False` — clean stop, right? But `ChatRunner._loop` calls `connector.is_connected()` every tick and on `False` prints "Meeting connection lost" plus the reconnection-attempt warnings from the Meet disconnect path. Visible as noise on the user's final `/quit`. Fix: route stdin `/quit`/EOF through `os.kill(os.getpid(), signal.SIGINT)` to trigger the existing SIGINT handler cleanly — same exit path as Ctrl-C. Generalizable: any new connector should distinguish "user asked to leave" (signal) from "connection died" (flag). Only the second one is a `_connected = False`.
- **`claude -p --resume <id>` looks up sessions by cwd, and must NOT be combined with `--worktree`.** (Session 126.) Setting up per-repo continuity in the delegate MCP meant pairing `--worktree` on first call with `--resume` on follow-ups. Two non-obvious traps: (a) running `claude -p ... --resume <id>` from the *parent repo* cwd errors with `No conversation found with session ID: <id>` — the session registry is indexed by project dir, and `--worktree` on first call changed the project dir to the worktree, so resume only works when cwd = the worktree path; (b) passing both `--resume` and `--worktree` on the same invocation does NOT reuse the original sandbox — `--worktree` spawns a brand-new one. Compounding: `claude --resume` with a bad session id exits 0 and writes the error to *stdout* as plain text, not JSON — so our JSON parser has to treat non-JSON stdout as an error path (not just stderr or non-zero exit). Fix: on resume, `cwd=worktree_path`, command has `--resume <id>` only, no `--worktree`. And on any non-JSON stdout containing "No conversation found", fall back to spawning a fresh `--worktree` call (handles the case where the user manually deleted the worktree dir between calls). Surfaced during verification in `/tmp/delegate_verify/` before committing the rewrite — first resume attempt from the parent repo failed silently from the MCP's POV (empty result), only the raw `claude` CLI run showed the "No conversation found" line.
- **`git diff` ignores untracked files by default — `land_worktree_changes` needs `git add -A` first.** (Session 126.) Claude Code's subprocess consistently leaves new files it creates **untracked** in the worktree (it only commits on explicit request). `land_worktree_changes` was computing `git -C <worktree> diff <base_sha>` to produce a patch for `git apply`, and the diff was empty even when the sandbox clearly had new files — because `git diff` (without `--cached` or `HEAD`) compares the working-tree version of *tracked* files against the index, and untracked files aren't in either. First landing attempt returned "No changes to land" for a sandbox containing an untracked `marker.txt` with content. Fix: `git -C <worktree> add -A` before the diff, then `git diff --cached <base_sha>` — staging pulls untracked files into the index so they appear in the diff. Side-effect-free because worktrees are throwaway sandboxes; staging them doesn't affect the parent repo's index or anywhere the user would notice.
- **Terminal harness pins `get_participant_count()` to 2 to force 1-on-1 mode.** (Session 125.) `ChatRunner` requires the `@brainchild` trigger phrase in rooms with >`ONE_ON_ONE_THRESHOLD` (=2) participants. A terminal session has one human and one bot — conceptually 2 participants. Returning 1 would trip the alone-exit grace timer. Returning anything ≥3 would force the user to type `@brainchild` before every message, breaking the test-drive UX. Pinning to exactly 2 is the sweet spot. Worth capturing because it's a non-obvious coupling between a "test harness" and the meeting-mode logic — future connectors (Slack? Discord? CLI chat?) all face the same decision.
- **MagicMock falls through `or` as truthy — mock the method or the `or` default never fires.** (Session 129.) Added `MCPClient.tool_timeout_for(tool_name)` and wired `chat_runner` to read it as `self._mcp.tool_timeout_for(tc["name"]) or config.TOOL_TIMEOUT_SECONDS`. Unit tests started failing with `TypeError: '<=' not supported between instances of 'MagicMock' and 'int'`. Cause: test mocks built via `MagicMock()` auto-create the new method as another MagicMock, which is **truthy**, so `or` short-circuits on it and returns the mock itself — which then fails integer comparison downstream in the heartbeat/timeout math. Fix: explicit `mcp.tool_timeout_for.return_value = None` in every fixture that builds a MagicMock'd MCP. Generalizable: any time you add a new method to an interface that's MagicMock'd in tests, every existing fixture needs an explicit stub for that method — MagicMock's "just works" default behavior is exactly what trips `or`/`and`/truthy-check fallbacks.
- **`idle_timeout_seconds` wasn't what it sounded like.** (Session 129.) Config had an `idle_timeout_seconds: 600` under `connector:` on every bot. Name implied "how long Brainchild waits in an idle browser before giving up." grep showed the constant was actually the max wait for the user to admit the bot from the Meet lobby (`connectors/macos_adapter.py:354` — a polling loop during the knock/admit phase). Two wrongs compound: the name doesn't describe the behavior, and power users adjust it to cover a different scenario than the one it controls. Fix: renamed to `LOBBY_WAIT_SECONDS` in `config.py`, updated both connector callsites. Lesson: any config key still wearing a name from an earlier architecture is tech debt — grep its reads when auditing, don't trust the name.
- **`git commit` without `-a` silently drops unstaged edits after `git mv`.** (Session 143.) - **Tool-confirmation prompts self-poisoned the LLM context when persisted as `kind="chat"`.** (Session 127.) `ChatRunner._request_confirmation` sends "I'd like to run X. OK?" into chat and `_send` appended it to the meeting record. On the next turn, `MeetingRecord.tail()` replayed it to the model as a prior assistant message — the model learned to mimic the harness's own wording as *plain text* before emitting a real `tool_call`. Symptoms: (a) the user saw two confirmation prompts per call (a hallucinated one from the model, then the real one from the harness) and had to type "ok" twice; (b) occasionally after the user's "ok", the model skipped the real tool_call and just hallucinated a result — caught during Phase 15.5.7 step 6 smoke when Engineer reported "README.md" as the contents of a repo that actually held `f.txt`. What made it tricky: no single line in the code was wrong — each piece (persist chat to record, replay tail to model, prompt for confirmation in chat) was correct in isolation; the bug was an emergent feedback loop from all three composing. Looked like a prompt-engineering problem at first (add a "don't parrot confirmations" nudge to system prompt) until reframed as "why are harness prompts in the LLM's prompt at all?". Fix: `_send` gained a `kind` kwarg; `_request_confirmation` passes `kind="confirmation"`; user replies while `_pending_tool_call` is set are also tagged `kind="confirmation"` in `_loop`. `pipeline/llm.py` already filtered `tail()` to `kind in {chat, caption}`, so one-word keyword change made the whole harness-plumbing layer invisible to the model while keeping the full audit trail in the JSONL. Generalizable: any text the harness *shows in chat for UX reasons* is not the same as a conversation turn — keep them in separate lanes from day one, or you'll get mimicry bugs that masquerade as model hallucinations.
