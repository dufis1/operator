## Working Style (follow for every step)

- One step at a time. Test and commit before moving on. Never batch steps.
- Before making any change: explain what you're about to do and why, in plain language.
- Smallest possible change. If a step can be broken down, do so.
- Don't touch anything out of scope. Note it, don't change it.
- Audience: technical product manager, not a senior engineer. Define terms when used.

**Two bugs caught in the smoke cycle, both fixed inline:**

1. **Rich markup swallowed `[x]`.** The checked-state marker for the bundled-skills toggle list rendered as a blank: Rich's markup parser treats `[x]` as a (non-existent) style tag and strips it. `[ ]` (with a space) is not valid markup so it printed literally — exposing the asymmetry only when some items were default-checked. **Design decisions made this session worth remembering:**
- **Port the observer JS verbatim; ignore the voice-loop 600-line tail.** Parity-first feedback is about preserving battle-tested code. The value in `voice-preserved/captions_adapter.py` is the `CAPTION_OBSERVER_JS` constant (lines 52-240) — scoped region selector, speaker badge logic, icon/system filtering, RAF batching. The rest was voice-coupling (wake detection, TTS echo guards, abort events, blocking `capture_next_wake_utterance()`) and a whole duplicate `MeetingConnector` subclass. Skipping that was correct; porting it would have been dead weight.
- **Unified single-stream transcript, tagged by `kind`.** Considered chat vs captions as separate streams and rejected: through the user lens ("what did Sarah just say about the deadline?", "summarize what we agreed"), the timeline reads naturally when speech and typed chat interleave in order. The LLM can distinguish via the `kind` tag + `[spoken]` prefix. Separate streams would be implementation hygiene at the cost of user-facing coherence.
- **Finalized utterances only on disk.** Deltas (streaming caption updates ~3/sec) stay in memory in the finalizer buffer; only the finalized utterance hits the JSONL. Keeps disk noise down, keeps `tail(n)` readable, matches the "one line per message" shape chat already uses.
- **Captions default off.** Chat is opt-in by nature (people type intending to address the bot). Captions ingest every participant's speech for the whole meeting. That's a meaningfully different consent surface, so the flag defaults `false`.
- **Wire captions in `__main__`, not `ChatRunner`.** `expose_function` must precede `page.goto()`. ChatRunner currently creates the `MeetingRecord` inside `run()`, which runs *after* `connector.join()`. Easier to create the record + finalizer in `__main__` (direct-URL path) and thread the record into `ChatRunner` via the existing `meeting_record=` kwarg than to refactor the lifecycle. Calendar-mode captions will need a separate pass (URL not known at startup).

## Hard-Won Knowledge (read before touching relevant code)

- **`rich.live.Live` + `Prompt.ask` don't compose; reach for `readchar` and a callable right-pane instead** (session 120). Building the picker primitive surfaced the architectural fork: Rich's `Prompt.ask` writes to the console synchronously and Live owns its render region, so they fight over the cursor + cause flicker if you try to host text input inside a Live block. The clean split is: picker phases own the screen via Live and read raw keys via `readchar.readkey()` (returns `\x1b[A` for arrows, `\n`/`\r` for enter, ` ` for space — handle both `\n` and `\r` for cross-platform Enter). Text-input phases (custom-mode name prompts, API-key prompts) drop the Live region entirely and use `Prompt.ask` normally. Trying to keep the build card "always visible" through text-input phases is a maze for very little gain — the picker phases are where the live vibe pays off; text input is short and sequential. Second design lever: the picker's right pane needs to support both per-choice previews (step 1 portrait gallery — preview is a property of the highlighted choice) and global state previews (steps 2/3 build card — preview reflects toggled state across the whole list, doesn't depend on cursor). One-knob solution: a `right_pane: Callable[[cursor, checked], Renderable]` argument that wins over per-choice `Choice.preview` when provided. The wizard passes `right_pane = lambda c, ch: state.card(mcps=names_where(ch))` — closes over global wizard state and re-renders on every keystroke without mutating state during render.
- **Rich markup parser swallows `[x]` silently — use non-ASCII glyph in brackets** (session 119). Rendering a checked-state marker in toggle lists with `[x]` made it render as a blank because Rich treats `[x]` as a (non-existent) style tag and strips it. The asymmetry only showed up when some items were default-checked; `[ ]` (with a space) printed literally because " " isn't a valid style name. Fix: switch to `[✓]` — non-ASCII content inside brackets isn't parsed as a style. Same applies to any character that *could* be a style name (`[u]`, `[i]`, `[b]`, `[red]`, etc). Reach for a glyph the parser won't try to interpret, or escape with `\\[` if you really need ASCII brackets.
- **`MeetingRecord.tail()` must scope to the current session or the LLM echoes prior runs' replies and stops calling tools** (session 117). `MeetingRecord` is append-only JSONL keyed by meet slug — same URL ⇒ same slug ⇒ same file. When a user reruns `operator <name> <url>` against the same meeting (common during development), the file accumulates every prior session's entries. `tail(n)` was originally implemented as "last n lines of the file regardless of origin," which meant `LLMClient._tail_messages()` replayed the agent's own prior assistant replies back to it as `role="assistant"` history. Downstream effect: the model pattern-matched on its prior outputs and replied directly instead of calling the MCP tool that originally produced those outputs. Worst case observed in diagnosis: user said "mojo jojo" → bot replied `Created https://linear.app/mojo-jojo/issue/MOJ-11/fold-my-laundry` with **zero tool calls logged** — it hallucinated the URL from MOJ-N patterns in prior replies. The tell is extremely easy to miss: fresh-slug runs (`operator <name>` with no URL, which hits meet.new and gets a fresh slug) work fine; only same-URL reruns regress. Symptom from the logs is `len(messages)=20/22/...` on the problematic runs vs `len(messages)=1` on the fresh-slug run. Fix: write a `{"kind": "session_start", "timestamp": ...}` boundary marker on every `MeetingRecord.__init__`; `tail()` scans back from the end of the parsed entries to the most recent marker and returns only entries after it. Raw JSONL preserves the full audit (you'll see two `"session_start"` lines in the file after a rejoin). Don't "fix" this by purging old history — the file is valuable as a durable record of the meeting; only the *LLM-visible window* needs to be session-scoped. Tests in `tests/test_chat_hardening.py::test_meeting_record_tail_roundtrip` cover the reopen-yields-empty-tail contract.
- **Startup banners must print before log output or they get buried; don't wait for MCP status if the loadout names are enough** (session 117). First attempt at the 15.5.4 startup banner called `_print_startup_banner(mcp, skills)` *after* `mcp_thread.join()` so the loadout line could show per-server ✓/✗ status. Live-verified on `operator pm`: the banner printed correctly but sat 40+ lines down in scrollback, below MCP tool-discovery chatter + `TIMING` lines + `meet.new resolved to ...` → user's first reaction was "where's the banner?" Even though the banner has blank lines above/below it, in a log stream it reads as just another log entry. Fix: print the banner as a *boot splash* — fire it immediately after `load_skills(config.SKILLS_PATHS)`, before any MCP connect or browser launch. Drop the ✓/✗ from the loadout line (the existing per-server MCP connect logs already surface status; startup failures are obvious from behavior). Show names only: `linear · github · 4 skills · claude-sonnet-4-5`. Rule of thumb: if a user-facing splash depends on info that takes measurable time to fetch, either print it twice (splash early + status line later) or drop the fetch-dependent part.
- **Playwright `sync_playwright.__exit__` can wedge forever on a dead CDP socket; removing `browser.close()` isn't sufficient; SIGKILL the Node driver to unblock it** (session 116). With `launch_persistent_context(headless=False, args=[..., "--headless=new"])`, Chrome sometimes self-exits a beat after the Leave-call click — the CDP socket dies but Playwright doesn't notice. Any subsequent teardown that sends a CDP request (explicit `browser.close()`, OR the context-manager exit's `playwright.stop()`) waits for a response that never arrives. Stack signature to recognize: `greenlet_main` → `_loop.run_until_complete(connection.run_as_sync())` → asyncio `_run_once` → `KqueueSelector.select` → `kqueue.control(None, ...)` with `timeout=None` (no scheduled work, waiting on I/O forever). Rough guide to reproducibility: flaky — roughly 1 in 3 tool-call-path shutdowns hit it, clean-path shutdowns almost never do. First instinct was "remove `browser.close()` since `sync_playwright.__exit__` handles teardown" — that helps the clean path but the same wedge signature reproduces inside `__exit__` on the tool-call path. The fix that actually works: force-kill the Playwright Node driver child process directly — that closes its socket, which unblocks the greenlet's `kqueue.select(None)`, which lets the asyncio loop exit, which lets the context-manager unwind. Implementation in `connectors/macos_adapter.py`: `leave()` does `_browser_thread.join(timeout=2)` → if still alive, `_kill_playwright_drivers()` (pgrep `-P <pid>` + match cmdline on `playwright` + `run-driver` + SIGKILL) → re-join 2s. Don't try to rely on Playwright's graceful teardown path when the CDP socket might be dead — just kill the driver.
- **ChatRunner's main-thread loop needs `_stop_event` guards after any blocking call to avoid a stray post-shutdown iteration** (session 116). `ChatRunner._loop()` runs on the main thread; SIGINT's shutdown handler runs on the main thread too, synchronously (setting `_stop_event`, then calling `connector.leave()` which blocks until the browser tears down). While the handler is running, the loop is frozen wherever it was — often inside `read_chat()` or `get_participant_count()`'s `result_q.get(timeout=N)`. When the handler returns control to the loop, the blocked call resumes and returns its result. The loop body then runs through to completion (logging participant count changes, etc.) *before* the top-of-while check re-reads `_stop_event`. Symptom: a stray `ChatRunner: participant count changed X → 0` log appearing several seconds *after* `MacOSAdapter: left meeting`, attributed to the drained queue returning 0. Fix: `if self._stop_event.is_set(): break` immediately after any blocking call in `_loop` (`read_chat`, `get_participant_count`) so the loop bails before processing results from a mid-shutdown unblock.
- **Google Meet leaves the "Admit N guest" pill widget sticky in the DOM after the guest is admitted** (session 115). The pill stays visible (`is_visible() == True`, not hidden, not detached) with the same "Admit 1 guest" text even after a successful tray click admits the knocker. Symptom: an admission verifier that waits for the pill to disappear loops forever; a polling loop that re-attempts whenever the pill is visible spams admission clicks (and any "log success unconditionally after action" code path emits false-positive `admitted via X` lines every poll cycle). Two cooperating fixes in `connectors/macos_adapter.py` admission block: (1) verify success via **participant-count delta** — call `_do_get_participant_count(page)` for up to 3s after the click; if it goes up, that's the real admission signal. (2) **Cooldown** keyed on `(pill_text, participant_count)` — if the next 2s tick sees both unchanged, silently skip; re-attempt only fires when text or count changes (new knocker, someone leaves). Don't try to detect admission via the pill itself.
- **`LLMClient` and the captions path can hold different `MeetingRecord` instances if you forget to call `set_record`** (session 101). When `__main__.py` constructs `LLMClient` *before* `meeting_record` exists (which is what the captions path does — record needs the URL, LLM doesn't), the LLM is left holding its default empty in-memory `MeetingRecord(slug=None)`. `ChatRunner.__init__` only auto-wires `llm.set_record(record)` *inside* the `if self._record is None` branch (chat_runner.py:66-72) — i.e. only when ChatRunner had to construct the record itself. Captions path pre-creates the record → ChatRunner skips the wire → LLM reads from the empty record → `_tail_messages` returns `[]` → Anthropic 400 `messages: at least one message is required`. Symptom: every `@operator` call fails the moment captions are enabled. Fix: explicitly `llm.set_record(meeting_record)` immediately after `meeting_record = MeetingRecord(...)` in `__main__.py` (line 218). Diagnostic tell in startup logs: two `MeetingRecord opened …` lines for the same run, one in-memory and one on-disk.
- **Google Meet keeps prior caption text visible in the same speaker's region; same-speaker continuations need fuzzy prefix dedupe** (session 101). Each finalize re-emits whatever Meet hasn't scrolled off yet, so without dedupe the JSONL accumulates `"prior + new"` for every continuation. Strict `text.lower().startswith(prior.lower())` is too brittle — Meet auto-corrects casing/punctuation on already-shown text between finalizes (e.g. `"here,"` → `"Here."`). The voice branch (`voice-preserved/pipeline/captions.py`) already had `_normalize_for_match` for the same problem (echo-suspect detection). Ported as `_strip_prior_prefix(text, prior)` in `pipeline/transcript.py`: tokenize both via `\w+` regex (case + punct insensitive), check token-level prefix match, cut the original text at the end-position of the last matched token. Per-speaker dict so A's prior never affects B. Test coverage in `tests/test_transcript_dedupe.py`. Edge case still uncovered: if Meet rolls the pane such that prior is no longer at the start (only its tail overlaps the new text's head), prefix match fails and duplicate words emit; would need longest-overlap search.

If writes are needed, swap to Grab's `cursor-talk-to-figma-mcp` (~25 mutation tools, requires Figma plugin + `bun socket` bridge, loses "paste a Figma URL" UX because it operates only on the file currently open in the plugin). Lesson: verify MCP server capabilities against source, not just README claims.
- **Thin Figma payloads are ground truth, not bugs (session 112).** First test URL returned 592 bytes with a single RECTANGLE node; felt like a truncation bug. Verified against Figma REST API directly (`curl https://api.figma.com/v1/files/<key>`) — the file genuinely contained one rectangle. Operator's anti-hallucination guard in the designer hints ("if get_figma_data returns a thin payload, say so plainly and ask for a different node-id — do not fabricate structure") fires correctly in this case. When Designer's critique feels unexpectedly shallow, verify the source frame actually has the structure the user expects before blaming the LLM.
- **Sonnet 4.5 over-triggers `load_skill` on casual prompts (session 112, tuning note).** "Gut check, what's the first thing you'd change?" fired `load_skill("design-review-feedback")` on first attempt — the test plan's own "fail signal" for T5. Redirect prompt ("brief thoughts?") switched the register correctly. Not a bug; a tuning knob. If it becomes a problem, tighten the skill's `description` field (the menu entry) to explicitly exclude casual/gut-check prompts, or raise the skill's bar via system prompt hint. Shipped as-is.
- **`figma-developer-mcp` spawns a grandchild process that survives top-level SIGINT (session 112).** MCP client sends SIGINT to the top-level subprocess (npx shim), which exits. The real `figma-developer-mcp` binary (grandchild via `npx`) doesn't receive the signal and lingers. Our existing `_kill_orphaned_children()` safety net catches it via `pgrep -P <our_pid>` and SIGTERMs it — so user-visible behavior is clean, but logs show "killed orphaned child". Not a regression; this is exactly why the safety net exists (see entry above on child process cleanup). Applies to any npx-wrapped MCP server.
- **`meet.new` URL-resolution trap: the interstitial URL ends with `/new` and has a query string (session 114).** Navigating to `https://meet.new` redirects through `https://meet.google.com/new?authuser=0&hs=178` before landing on the real room `https://meet.google.com/<xxx-xxxx-xxx>`. A lambda like `u.endswith("meet.google.com") == False and "/new" not in u` is not enough — query strings bypass naive path-ending checks. Fix: use `urlparse(u).path` to strip query/fragment and validate with a regex `^/[a-z]{3,}-[a-z]{3,}-[a-z]{3,}/?

 (three hyphen-separated lowercase groups is Meet's room-code shape). Used as the predicate to `wait_for_url`. Symptom when this breaks: the *bot* lands in room A and `webbrowser.open(meeting_url)` hands the human the interstitial URL, which causes Google to spin up a *different* fresh meeting under the human's account — two rooms, no overlap.
- **Google Meet's "Admit N guests" pill is a hover-tray widget, not a button (session 114).** The visible green pill in the top-right when someone knocks is *not* a `<button role="button">` — it's a nested div with aria description "Press Down Arrow to open the hover tray and Escape to close." Clicking the pill element directly doesn't fire Meet's admit handler; you have to open the tray first (via hover or keyboard) and then click the `Admit` button that appears inside the tray. Synthetic `dispatchEvent(new MouseEvent('mouseenter'))` does NOT open the tray — Google's widget is bound to real mouse events. Playwright's `.hover()` does work because it moves the cursor via CDP (real browser-level mouse event). Keyboard path (`pill.focus()` → `ArrowDown` → `Enter`) is the fallback since the aria description explicitly documents it.
- **Playwright `:has-text()` matches ancestors, not just leaves (session 114).** A selector like `:is(button, [role="button"]):has-text("Admit")` seems scoped but actually matches *every ancestor* whose descendant subtree contains the text "Admit" — including body-level containers. In our first auto-admit attempt this over-matched ~20 unrelated elements per poll cycle and (accidentally) triggered unrelated UI like reopening the chat panel. If you need a tight "element whose accessible name IS X" match, use `get_by_role("button", name=re.compile(r"^Admit$", re.I))` instead; `:has-text()` is for content searches across subtrees.
- **Already-in-meeting detection via `Leave call` button (session 114).** When the bot creates a meeting via `meet.new`, Google auto-joins it as the creator — by the time the adapter's pre-join flow starts, the "Join now" / "Ask to join" / "Sign in" buttons don't exist because we're *already in*. The pre-join race times out on all three and the adapter tore down with `join_fail`. Fix: before the pre-join race, probe for `get_by_role("button", name="Leave call")` — if visible, set `clicked_label = "already_in"` and skip both camera toggle and the join-button race. `Leave call` is a reliable "am I in the meeting?" oracle because it only exists during an active call.
- **The bot must stay headless; the human needs a separate visible tab (session 114).** The bot's Chrome runs with `--headless=new` — that's intentional (invisible observer). Don't remove `--headless=new` to "show the browser" — that makes the bot visible, which is wrong. Instead, when the bot auto-creates a meeting (meet.new path), pop a *second* visible tab in the user's default browser via `webbrowser.open(meeting_url)` from `__main__.py`. The bot and human connect to the same room through two different browser sessions. Only call `webbrowser.open` after the room URL is resolved (never pass the interstitial).
- **Rich's markup parser silently swallows `[x]` in toggle lists (session 119).** When rendering a numbered checkbox list in the setup wizard via `rich.console.Console.print()`, `[x]` disappears from output while `[ ]` (space inside) prints literally — asymmetric and easy to miss. Rich treats `[...]` as a style tag; `[x]` matches the "nonexistent style `x`" grammar and gets stripped, but `[ ]` doesn't parse as a valid tag so it survives. Symptom: first smoke-test of `operator setup` showed bundled skills as `1. · alpha` instead of `1. [x] alpha`. Fix: use a non-bracket glyph for the "on" mark — `[✓]` for checked, `[ ]` for unchecked. Alternative would be `Console(markup=False)` but that kills other useful formatting. Rule of thumb: anything enclosed in `[…]` that isn't a valid style name is at risk; prefer Unicode glyphs for user-facing status indicators in Rich output.
- **Edit-in-place bundle rewrites must seed the tempdir with the existing bundle first, or unknown files get wiped (session 119).** The `operator setup` wizard uses atomic write — build into `agents/.<name>.tmp-<ts>/`, then `os.rename` it over `agents/<name>/` (with a `.bak-<ts>` swap for recovery). First implementation built the tempdir from scratch with only the wizard-owned files (`config.yaml`, `portrait.txt`, `skills/`, stub `README.md`), then swapped. Result: `.env.example`, hand-written README edits, delegate scripts, and any other non-owned siblings in the preset bundle got nuked silently on reconfigure — the user never consented to that. Fix: in `edit` mode, seed the tempdir via `shutil.copytree(target, tmp, dirs_exist_ok=True)` *before* any writes so the full existing bundle carries through the swap. Then selectively overwrite owned files, wipe `skills/` wholesale (skills are re-authored each run), and write `README.md` only if absent. The test `test_edit_in_place_swap_cleans_backup` pins this: asserts `.env.example` + pre-existing README survive, stale `skills/deselected/` is gone, no `.bak-*` lingers on success. Generalizable rule: atomic-write flows that "replace a directory" must preserve any files the operation doesn't explicitly own.

## LLM Interaction Principles

Design patterns for working with LLMs in this product. Read before making architectural decisions about context management or model behavior.

- **Archive-with-metadata pattern:** When removing content from the model's context (tool results, history turns), always replace it with a placeholder that names (1) what happened and (2) what the model can do about it. Example: `[tool result archived — call the tool again with a narrower scope to retrieve more]` is better than silent deletion or `[result omitted]` (which implies error). The model reasons better when it understands its own state and has a recovery path. Applies anywhere context is managed: history trimming, tool result truncation, overflow recovery.

## Phase 11.4 — Skill Loading (design notes)

**History (session 102).** Originally split as 11.4 (generic flat-`.md`) + 11.5 (Claude Code interop on top). Collapsed into a single phase: adopting `SKILL.md` shape + progressive disclosure + `allowed-tools` filtering up front is simpler than shipping twice. `skills.local.yaml` deferred to post-MVP.

Pitch: **"Claude Code in Google Meet."** Users bring their existing Claude Code skills into Operator instead of recreating them.

**Source of truth.** Claude Code skills live at `~/.claude/skills/<name>/SKILL.md` (user-level) and `.claude/skills/<name>/SKILL.md` (project-level). Each skill is a folder with a `SKILL.md` — YAML frontmatter (`name`, `description`, optional `allowed-tools`) plus a markdown body. Format is designed for progressive disclosure: only `name + description` loads up front; the full body is pulled in when the model decides the skill is relevant.

**Load model: reference, not copy.** `config.yaml` stores paths (`~/.claude/skills/<name>` or absolute paths), not file contents. On startup, Operator reads `SKILL.md` live from the user's machine. Rationale: each user's `~/.claude/skills/` is already their curated, personal set — forcing a copy into the repo would be double-bookkeeping, and the open-source story is "each user makes Operator theirs." Downside acknowledged: paths only exist on that user's machine, so a teammate who clones the repo will hit missing skills. Mitigation: log + skip, don't crash.

**Implementation shape.**
- `config.yaml` gains a `skills:` list. Entries are paths (absolute or `~`-expanded). An entry may be either a single skill folder (contains `SKILL.md`) or a parent directory (scanned one level deep for `*/SKILL.md`). Parent-dir scanning is what makes `~/.claude/skills/` a single-line config entry.
- On startup, resolve each path, parse `SKILL.md` frontmatter with `pyyaml` (already a dep).
- **Progressive disclosure.** Inject only `name + description` of each skill into the system prompt as an available-skills list. Register a synthetic `load_skill(name: str)` tool alongside MCP tools. When the model calls it, Operator returns the full body of the named skill as the tool result. Model then re-plans with the body in context. Matches Claude Code's loading behavior; keeps token cost flat as users accumulate skills.
- Missing path → WARN log + skip. Do not crash startup.
- Malformed `SKILL.md` (bad YAML, missing `name` / `description`) → WARN log + skip.
- Duplicate skill names across paths → last one wins, WARN on collision.
- Startup log line: `SKILLS: N/M loaded (<name> ✓, <name> ✓, <name> ✗ missing)`. Mirrors the MCP startup banner pattern from session 87.
- **Deferred to post-MVP:** optional gitignored `skills.local.yaml` for personal overrides — teams share a baseline `config.yaml` and layer personal skills on top. Not MVP-critical.
- System prompt ordering: base prompt → skills block → MCP hints → MCP status (most dynamic last, so skills share a stable prefix for prompt caching).

**Tool-surface filtering.** Most interesting Claude Code skills assume tools Operator doesn't have in a meeting context. Split by `allowed-tools`:
- **Pure-instruction skills** (no `allowed-tools`, or only MCP-backed tools): ship as-is. Tone guides, commit-message style, decision frameworks, interview rubrics.
- **MCP-backed skills** (Linear, Gmail, Calendar): work once the corresponding MCP server is configured.
- **Tool-heavy skills** (`Bash`, `Edit`, `Write`, `Read`): known to misfire. Five common ones that break: `/commit`-style git skills, the `pdf` skill, the office-suite skill (`docx`/`xlsx`/`pptx`), PR-review skills (`/review-pr`, `simplify`), refactor/debug-test-failure skills. Common thread: **anything that assumes a filesystem, a shell, or the Read/Edit/Write triad** breaks in a Meet context.
- On import, check `allowed-tools`; if it references tools Operator can't honor, log a WARN with the specific unsupported tools. Still load the skill (the model may decline to invoke it in the meeting context) but the user knows why it might misfire.

**Out of scope for 11.4.** Skill *execution* beyond injecting body text — i.e., we are not implementing `Bash`/`Edit`/`Write`/`Read` to make tool-heavy skills actually work. Pitch is honest: "bring your instruction skills." Also out of scope: hot-reload when a `SKILL.md` changes on disk mid-meeting — load at startup only (restart required for changes).

## Phase 15.5 — Opinionated Quickstart (design notes)

Pitch: **"Choose your fighter → add power-ups → go."** The agents gallery is the gallery layer; the wizard turns it into a 3-step setup. Implicit differentiator vs. joinly's framework-shaped positioning.

**Why a gallery, not forks.** Forks fragment git history and don't make the parent repo look alive. A `roster/` folder with PR contributions keeps everyone on main, makes contributors visible in the commit log, and turns every seeded-persona build into a permanent artifact. This is the primitive; forks were rejected explicitly. Reference patterns: Ollama model library, LangChain templates, n8n templates, ComfyUI workflows, Vercel templates, Hugging Face Spaces.

**Folder format (already locked in `roster/README.md`).** Each roster member is a self-contained folder with a runnable `config.yaml`, optional `skills/` directory, a short `README.md` (what / who / needs / setup / demo), and `.env.example`. The roster member must run on a fresh clone after the user fills in keys — no hidden dependencies.

### 15.5.1 — `claude-code` starter agent

Canonical example. Demonstrates the format and is the launch hero. **Depends on Phase 11.5 (Claude Code skill import) being complete** — without skill import, this agent has nothing distinct to show.

- `roster/engineer/config.yaml` — complete Operator config; either OpenAI or Anthropic as default (decide near build time, lean Anthropic since the framing is "Claude Code"); `mcp_servers` includes Linear + GitHub; `skills:` points at `~/.claude/skills/` by default, with a comment explaining how to swap to a custom path.
- `roster/engineer/README.md` — "What it does: brings your existing `~/.claude/skills/` into Google Meet so the agent files Linear tickets, looks up GitHub PRs, and follows your team conventions during a live call." Setup: copy `config.yaml`, fill `.env`, run. Demo: short GIF of an artifact appearing in chat mid-sentence.
- `roster/engineer/skills/` — leave empty by default. The whole point is the user's own skills get loaded from `~/.claude/skills/`. Optionally bundle one demo skill that's safe to ship (e.g. `file-linear-ticket.md`) so the agent has at least one skill even if the user has none locally.
- `.env.example` — `OPENAI_API_KEY` or `ANTHROPIC_API_KEY`, `LINEAR_API_KEY`, `GITHUB_TOKEN`.

### 15.5.3 — Runtime architecture (roster as source of truth)

**The architectural shift** (session 113): today, `python __main__.py <url>` reads an ambient `config.yaml` at repo root — "whatever was last copied there." Replaced by explicit per-run bot selection. Every run names the bot.

- Positional CLI in `__main__.py`:
  - `operator <name> <url>` — run that bot in that meeting. Load `roster/<name>/config.yaml` directly.
  - `operator <name>` (no URL) — auto-open `meet.new`, capture slug, join as that bot. (Absorbs old 15.5.4's `meet.new` deliverable.)
  - `operator setup` — launches the wizard (see 15.5.5).
  - `operator list` — prints available roster members + one-line pitches.
  - `operator` (bare) — prints usage + roster list.
- Dispatch rule: if first arg matches a `roster/*` folder, treat as bot-name; if it's a known subcommand (`setup` / `list`), dispatch there; else error with helpful usage.
- Delete root `config.yaml`. Config loading reads `roster/<name>/config.yaml` at runtime.
- `.env` stays at repo root, shared across all bots (keys like `ANTHROPIC_API_KEY` are environment-scoped, not bot-scoped).
- Ship `./operator` bash wrapper in repo root: `#!/usr/bin/env bash` + `cd "$(dirname "$0")"` + `source venv/bin/activate` + `exec python __main__.py "$@"`. Provisional launcher until product name is locked and curl installer ships post-launch.
- **Breaking change:** every doc, test, and reference to `python __main__.py <url>` migrates to `operator <name> <url>`. Pre-launch, so the cost is grep + update, not deprecation.

### 15.5.4 — Face system + profile banner

Each bot has a unique pixel-art face that prints on every run. Faces are the bot's identity and the "you're playing as X" moment when a meeting starts.

- **Deterministic face generator.** Half-block Unicode (`▀ ▄ █`) + ANSI 256-color. 16×20 pixels — fits 3-across in a standard terminal, readable. Seed = hash(bot name), so the same name always produces the same face. Parts library: eyes, mouth, brow, hair/hat, accessory, palette.
- **Shipping portraits.** Run the generator for `engineer` / `pm` / `designer`, hand-curate until good, commit as `roster/<name>/portrait.txt`.
- **First-run write hook.** Any `operator <name>` run that finds no `portrait.txt` in the roster folder generates one and writes it. This is the "gift" for contributors: they add `roster/translator/` with a config and README, run `operator translator` once, and a face file materializes for them to commit with their PR.
- **Startup banner.** Every `operator <name>` run prints a block before joining:
  ```
  ╔════════════════════════════╗
  ║   <face>      ENGINEER     ║
  ║               looks up GH  ║
  ║               MCPs: GitHub ║
  ║               Skills: 4    ║
  ║               Model: ...   ║
  ╚════════════════════════════╝
  ```
  Reads from the active config — pulls agent name, one-line tagline (from README h1 or a config field), MCP names, skill count, model.
- **`--plain` fallback.** Prints ASCII-only banner for screen readers / hostile terminals / CI logs.
- Dep: `rich` for color/boxes/layout (new dep — already battle-tested, 1-package install).

### 15.5.5 — `operator setup` wizard (creation flow)

**Key reframe** (session 113): wizard is no longer "activate a bundle by copying config to root." It's a *creation* tool — builds a new `roster/<name>/` folder so the user ends up with their own bot they run explicitly.

- `operator setup` — four steps:
  1. **Choose a template (or start blank).** Fighter-select grid UI — colored boxes, pre-shipped faces visible, numbered prompt. "Start blank" produces a minimal config stub.
  2. **Power-ups — awareness + toggle.** Show the template's bundled MCPs and skills as a checklist (including commented-out power-up entries in the template config). User toggles what ends up in the new bot. Narrow scope: only touches what the template declares. Out of scope: arbitrary MCP discovery, paste-a-URL flow, `~/.claude/skills` path editor (all post-launch).
  3. **Keys.** Prompt for any env vars the new bot needs that aren't already in `.env`. `getpass` for secrets (no echo, never print existing values). Merge into `.env`, preserving unrelated keys. Atomic write.
  4. **Finalize.** Atomic-write the new `roster/<new-name>/` folder: `config.yaml` (built from template + toggles), `README.md` stub, `.env.example`. Generate face via 15.5.4's generator, write `portrait.txt`. Print: `Your bot is ready. Run: operator <new-name> <meet-url>`.
- Re-run-safe: name-collision check against existing roster folders.
- Intent of step 2: pure FYI — user should *know* what their bot can do (so they know to ask it to file a Linear ticket), not be forced into configuration work. Checkbox ergonomics, not a forms-based MCP configurator.
- **Depends on 15.5.3** (for `setup` dispatch surface) and **15.5.4** (for face generator + pre-baked shipping portraits rendered in step 1's grid).
- ~250 lines of Python + `rich`. Out of scope: voice, MCP OAuth flows, model selection UI beyond what the template dictates. Those land in the post-MVP full wizard.

### Launch interplay

The gallery doubles as launch ammunition (see `docs/mvp-bar.md` Launch Strategy). Each seeded persona contributes one `roster/<name>/` PR, then posts about that build. Real users browsing the repo see contributor activity on `main` — social proof as code. Ship 3 roster members at launch (engineer + PM + designer), target 5 within week 1 via seeded PRs.

## Hard Won Knowledge

*Append-only log of bugs that were hard to diagnose. Each entry: what the bug was, what made it tricky, how it was fixed.*

- **`yaml.safe_dump` silently eats comments, blank lines, and hand-formatting.** (Session 121.) The setup wizard's edit-in-place mode runs `yaml.safe_load(existing_cfg)` → user edits → `yaml.safe_dump(cfg)` → writes back. Comments, section dividers, inline explanatory notes, and custom quoting all vanish. Surfaced when the user smoke-tested the wizard on `agents/{pm,designer,engineer}/config.yaml` — `git status` showed huge diffs even on presets that didn't change any wizard inputs, just because `yaml.safe_dump` round-trips drop everything the parser doesn't care about. What made it tricky: the wizard behaves correctly (values round-trip intact) so no test would catch it; you only see the damage in `git diff`. Mitigation for now: restored configs via `git checkout --`. Proper fix (deferred): swap to `ruamel.yaml` with `YAML(typ='rt')` — round-trip loader preserves comments and formatting. Or, only rewrite the specific fields the wizard touches (`enabled` flags, `skills.paths`) via a targeted string/diff approach. Not blocking; worth addressing before inviting external contributors who'll keep editing their configs by hand.
- **Rich markup parser swallows `[x]` in printed text.** (Session 119.) First draft of the power-ups toggle rendered checked items as `[x]` and unchecked as `[ ]`. `[ ]` printed literally (invalid markup), but `[x]` vanished entirely — Rich interpreted it as a (non-existent) style tag and stripped it. Took a minute to spot because the asymmetry only appeared when some items were default-checked. Fix: switched to `[✓]` (non-ASCII bracketed content isn't parsed as a style tag). Generalizable lesson: any `[letter]` or `[word]` in Rich-printed strings is at risk of being treated as markup — either escape with `\[` or use unicode glyphs.
- **Rich `Live` can't host `Prompt.ask`/`input()` mid-render.** (Session 121.) First attempt at step 3's "add your own skills" UX tried to pop a text input when a sentinel row was toggled mid-picker. Doesn't work — the `Live` region owns the terminal until exited, and input prompts don't compose with it. Fix: either exit the Live first (picker confirms → then prompt), or render a "queued action" hint and act on it after the picker closes. Final design landed on the former — `Confirm.ask` after the picker closes, path-input loop if confirmed.
- **Right-pane column squeezes inline sublabels past terminal-width truncation.** (Session 121.) Picker rows rendered `label <spaces> sublabel` on one line in a `Table.grid` left column; a 40-col build card in the right column left only ~32 cols for the left, and Rich silently truncated the dim sublabel text with no ellipsis. Looked like a 60-char cap bug at first because the skill subtitle *also* had a `desc[:60]` truncation in the code, which was a red herring — removing the slice didn't fix the visible cut-off. Fix: move sublabels to their own indented line under the label. No more horizontal-width race.
